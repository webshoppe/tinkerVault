# Project Summary, Whiteboard App

This is the story behind the tool, not the tool itself. If you just want to use it, `README.md` is right here in the same folder.

This document has two voices, kept deliberately separate rather than blended into one: the agent that actually wrote the code, and an outside narrator who watched the whole three-tier build happen and did the independent verification. They don't always agree on emphasis, and in one place they don't fully agree on a finding. That's kept intact on purpose. If you're on a similar quest, both pathways of reasoning are probably more useful to you than a single flattened account would be.

---

## Origin and setup

**From the outside.** This idea was a deliberate deferral, surfacing during a brainstorm for a completely different, unrelated build (a markdown viewer, see that project's own `PROJECT_SUMMARY.md` if you're curious), set aside as too ambitious for a first test, and picked back up here on its own, in a different tool entirely. The brief going in, in the project owner's own words: "I almost like the whiteboard, but want to really enhance that and polish it and make it really pop, but be so extremely useful, its comical." That's a real design target, not just enthusiasm, it shaped every scope decision below.

**The agent's account.** The goal never changed much: a portable clipboard/whiteboard that opens as a single HTML file, needs no install and no server, and still feels like a real app, not a demo. Scope was deliberately split into three tiers so Tier 1 could prove the architecture before multi-board and packaging piled on. Environment context mattered more than it should have: development lived in WSL2, but real browser verification had to be forced onto Windows Chrome (and later Playwright Firefox/WebKit on Windows) because Linux Chromium inside WSL was missing system libraries, with no `sudo` available to fix them. The runtime product still has zero Node dependency; Node and Playwright only exist for the checks under `build-process/`. That split, fat single-file product, thin optional test harness, is the spine of the whole project.

*(Neither account is more "true" here, they're describing different layers of the same build: why it exists, and how it was actually exercised day to day. Worth knowing both.)*

## Why Tier 1 was really two things at once

**From the outside.** The stated goal was an app shell, a pluggable template architecture, and two working templates to prove it. The unstated second goal was calibration: there was zero prior data on how this specific agent behaved across a long, many-tool-call, fully unattended run. That shaped a real decision: which permission mode to run it in. The tool's own docs are explicit that only two of its four session modes are actually confirmed to enforce anything. Given the actual blast radius, a brand-new empty folder, no git, no credentials, nothing else on the system in reach, the fully-autonomous option won, on the condition a human was watching the terminal periodically rather than walking away. It held up. No destructive or out-of-scope action of any kind. Independent human verification followed anyway, drawing, sticky notes, and specifically a hard refresh (not just a normal reload) to rule out the possibility that persistence was quietly riding browser cache instead of actually reading from storage. It wasn't. Total cost: about 1% of a weekly usage allowance.

**The agent's account.** Even with only Paint and Notepad shipping, the shell was built around one contract: `register → create(ctx) → setState / mount / renderTools / getState / unmount`. That paid off immediately in Tier 2, Annotate and Wordpad were new blocks of code, not a rewrite of the shell. Multi-board became "each board stores a `templateId` plus opaque `state`," which is almost boring, and boring is good when you're also fighting `file://` and localStorage quotas. If starting over, that contract would still get invented on day one. Hardcoding two canvases would have made Tier 2 a fork-and-rewrite mess.

## Tier 2: multi-board, Annotate, and a test that caught itself

**From the outside.** The single most consequential addition wasn't a template at all, it was letting the app hold more than one saved board, with the ability to create, rename, duplicate, and delete them. One specific design worry going in turned out to be unfounded: the expectation, based on a different constraint hit during the unrelated markdown-viewer build, was that real screen capture probably wouldn't work from a plain double-clicked local file either. In manual testing across two real browsers, it did, no workaround needed. The lesson isn't "that constraint doesn't exist," it's that constraints from one browser API don't automatically transfer to a different one that merely sounds similar.

**The agent's account, on the same feature, with more nuance.** Treating "multiple named boards" as the main Tier 2 feature, not "more drawing tools," was the right call. Storage moving from v1 (a map of template states) to v2 (an array of boards) was a real schema break; writing an explicit migration from `whiteboard-app-v1` to `whiteboard-app-v2` avoided stranding anyone who'd already used Tier 1 in the same browser origin. On screen capture specifically, the framing is more careful than "it just worked": the decision was clipboard paste, drag-and-drop, and file open as the *primary* Annotate workflow, with the capture button present only when the API exists and failures surfacing a status message pointing back to paste and drop. The reasoning: `file://`, permission UI, and headless verification can't all reliably agree with each other, so shipping "a screen grabber that sometimes does nothing" as the main path would have been the wrong failure mode, regardless of whether it happens to succeed in any given session. That's a meaningfully more hedged claim than "it worked fine," worth sitting with both versions rather than picking one.

**On the test bug, two levels of the same story.** From the outside: the agent's own verification suite initially failed one check related to undo/redo, then fixed its own test and re-ran clean, worth extra scrutiny rather than taking at face value, since a "test fix" can mean the test was wrong or that a real bug got quietly papered over. A manual undo/redo stress test, multiple undo steps then multiple redo steps, confirmed it was the former. The agent's own account of exactly what was wrong: an automated check asserted undo by comparing string *lengths* after typing. The test had left the editor with a full selection (after bold plus select-all), so typing replaced the whole document with a short string, and undo correctly restored the longer original. The feature was right, the assertion measured the wrong proxy. Takeaway from the agent itself: UI tests that measure length or count instead of actual content will gaslight you.

## Tier 3: packaging, and a bug neither account would have caught alone

**From the outside.** The last tier was deliberately scoped to add nothing new. The one substantive ask beyond packaging was verifying across multiple browser engines, not just the one used throughout earlier tiers. That paid off directly: cross-engine testing surfaced a real, engine-specific bug, invisible in the engine used for every earlier check. One irreducible gap remains: one of the three engines tested has no native counterpart on the machine doing final review, so there's no independent way to double-check that engine's result by hand, it rests entirely on the automated tool's own report.

**The agent's account of the actual bug.** Cross-engine runs passed Chromium and Firefox, then WebKit failed with `TypeError: null is not an object (evaluating 'c2d.save')`. Switching boards unmounted Paint while a `ResizeObserver` and a delayed redraw were still running and touched a null canvas context. Chromium's timing simply didn't hit it as often, which is exactly why single-engine verification would have shipped this. Fix: an `alive` flag, canceling pending resize timers on unmount, and null-guards on canvas operations. Takeaway, in the agent's own words: multi-engine testing isn't for the screenshot gallery, it's for lifecycle races, single-page apps with `unmount` need the same paranoia as cleanup functions in any component framework.

There's also an honest archival note worth surfacing: Tier 1 was verified via chat output and screenshots, but the JSON report file on disk was later overwritten by the Tier 2 suite, so no separate Tier 1 report survives on disk, only the narrative and screenshots. Lesson for next time, version report filenames per tier from day one. Tier 3 already did.

## What the agent was unsure about (a section with no outside counterpart)

This part only exists from the inside, an outside observer can verify *that* something works, but not always *why* the builder chose one approach over another.

- **How far to push "single file."** Keeping CSS and JS inline makes distribution trivial and code review harder, there's no module graph, only section comments and scroll distance. For a 1.0 clipboard app that must never depend on a server, the tradeoff still felt correct. For a much bigger fifth or sixth template, someone will eventually want to split for editing and concatenate for release, not needed by 1.0.
- **Wordpad without a framework.** `contenteditable` plus `document.execCommand` is legacy and uneven, but it works offline with no library. A heavier editor would have violated the no-CDN, no-install constraint or forced vendoring thousands of lines. The bar accepted was "good enough Wordpad," not a rich editor.
- **The undo model.** Paint and Annotate undo are snapshot stacks of PNG data URLs; Wordpad is HTML snapshots with a debounce. Memory-heavy and not stroke-perfect, but simple, and it serializes for free with the rest of the board state. A vector stroke history would have been cleaner long-term and more work than the tier allowed.
- **Whether multi-engine verification was worth the ceremony in packaging.** It nearly felt like overkill, until WebKit failed once. One failing engine is enough to justify the matrix.

## Decisions worth naming explicitly

- **`getDisplayMedia` as bonus, never backbone**, for the reasons above.
- **Fixed template type per board.** You choose the type at creation; you don't transmute a board later, avoiding conversion edge cases like "what does turning a Wordpad board into a Paint board even mean for its state."
- **JSON import appends, never silently replaces.** A safer default next to a destructive "replace my whole library," users can delete after import if replacement was actually intended.
- **PNG-in-localStorage for Paint and Annotate.** Chosen for fidelity and simplicity, at the cost of quota pressure and larger export files. The status bar reports save failures instead of failing silently, that part was treated as non-negotiable.
- **No LICENSE file shipped.** Not a technical call, a deliberate deference to the project owner's own decision to make later.

---

## If a sibling app got built tomorrow (the agent's own list)

1. Contract-first shell (registry plus state) before any second feature.
2. Design the primary offline path first; treat privileged browser APIs as optional from the start, not an afterthought.
3. Per-tier report filenames, committed or copied into `build-process/` as you go, not reconstructed after the fact.
4. At least one second browser engine before calling UI lifecycle work "done."
5. Keep the double-click HTML file as the only runtime artifact anyone has to understand.

## A process note the outside account adds

The permission-mode decision from Tier 1 didn't get treated as a standing default once it worked. Each tier's scope got a fresh, brief check: does this still only touch a self-contained project folder, still nothing involving git, credentials, or the wider system. It did, every time, so the same autonomous approach carried through all three tiers without incident, but that check happened each time, not once. A verdict earned under one scope doesn't automatically transfer to a bigger one.

## Part two: v2 (2026-07-12), five more tiers, then packaging

Same format continues below: outside account first, agent's account second, kept in the same two separate voices as Part one. 1.0.0's own bottom line (further down) has been rewritten to cover the whole arc rather than stopping at three tiers; nothing above this line was altered.

**From the outside.** The brief for v2 picked up where 1.0.0 left off: not "did it work," but "how much more useful can this get without breaking what already works." An unfiltered brainstorm ran to 65 ideas across ten categories before any scoping happened. The cut that survived leaned toward that doc's own read of "highest usefulness per unit effort" plus outstanding v1 debt (the known localStorage quota risk, no corrupt-state handling, no multi-tab safety), with three challenges deliberately raised and settled before the plan locked: sequence the quota fix ahead of anything that adds more storage pressure (the time machine, specifically, since snapshots make the exact problem the quota fix addresses worse if it ships first); pick one paradigm for smart paste rather than shipping two competing UX models (auto-route the unambiguous cases, toast-suggest only the genuinely unclear ones); and don't skip the cheapest, highest-value heuristic wins (auto-checklist, auto-color, auto-tags) just because they're less flashy than search or a new template. All three held through to the final five-tier scope.

**The agent's account, tier by tier** (full narration for each lives in that tier's own `STATUS.md` in the working build folder; this is the throughline):

- **v2 Tier 1** (insurance + one power feature): a visible quota meter with bitmap auto-downscale past a storage threshold, schema-validated corrupt-state recovery that quarantines bad data and falls back to a clean board instead of white-screening, a `storage`-event banner for multi-tab conflicts (autosave paused until the user picks a side), global search on `Ctrl/Cmd+K`, and full session restore (board, zoom, scroll, last tool per template). 53/53 on real Chromium.
- **v2 Tier 2** (heuristic auto layer): auto-checklist detection on sticky-note text, keyword-driven auto-color with a user-editable keyword map, `#tag` extraction with a click-to-filter tag bar, and a command palette deliberately bound to `Ctrl/Cmd+/` rather than reusing Tier 1's `Ctrl/Cmd+K`, specifically to avoid rebinding already-shipped, already-tested behavior. 57/57.
- **v2 Tier 3** (paste intelligence): the app now routes paste and drop by content. Silent auto-route for unambiguous input — images to a new Annotate item, `.txt`/`.md` to a new Wordpad page, a pasted string that's just a URL to a domain-only link card with no network fetch (fetching a real page title would break the offline-only design), short text (≤280 chars) to a sticky, long text (≥900 chars) to Wordpad. A dismissable toast for genuinely ambiguous cases: text in the 281–899 character band, tabular-looking paste (offered as a Wordpad table, there's no dedicated spreadsheet template), and a `.json` drop offering a board import, never silent. First pass landed 51/51 clean, but manual testing found the routing looked broken at lengths nowhere near the documented thresholds. Root cause, once investigated, was not the thresholds — 280/900 were correct in the code the entire time. It was a stale toast from an earlier ambiguous paste not clearing before the next one, plus a deliberate design choice (pasting with focus already inside an open Wordpad document skips smart routing on purpose) that read as "silent long-text routing" for a length that should have gone to a sticky. A bonus find in the same pass: tabular detection was false-positiving on ordinary prose with one comma per line. All three fixed, thresholds re-verified in real Chromium at five concrete lengths, full suite re-run at 68/68.
- **v2 Tier 4** (new template): Kanban, the first new template since the original four. Editable columns (three defaults: To Do / In Progress / Done), live WIP counts, cards that reuse Tier 2's auto-color/checklist/tag detection on their text, and two non-destructive promotion paths — a single sticky, or an entire Sticky Notes board, into a new Kanban board, with the originals always left untouched. First pass reported 31/31, but dragging a card between columns didn't work at all by hand. Root cause: the card's draggable hit area was mostly a `<textarea>`, which blocks a native HTML5 drag from starting when grabbed anywhere in the body text in Firefox; separately, the verify script had been moving cards by calling the internal state-update function directly rather than driving an actual pointer gesture through the DOM, so a completely non-functional drag interaction still reported clean. Rebuilt on a dedicated `⋮⋮` grip handle (the text area stays reserved for editing), drop-target detection via `elementsFromPoint`, and a rewritten test that drives real `pointerdown`→`pointermove`→`pointerup` sequences plus a reload check that the new column sticks. 39/39 on re-verification.
- **v2 Tier 5** (history, the last tier of the agreed scope): a rolling window of up to 10 snapshots per board, cadence-limited rather than fired on every autosave tick, with forced snapshots before a restore or a trash action so neither can silently discard the current state; a restore picker with a timestamp and a lightweight text summary per snapshot; a 7-day soft-delete trash for boards with auto-purge and a standing refusal to remove the last live board; and a quota meter that now counts history and trash storage, not just live boards, trimming the oldest snapshots first under pressure before falling back to Tier 1's more aggressive image compression. This time the verify suite was deliberately written to drive real menu clicks and confirmations end to end, rather than a direct state call, learned directly from the previous two tiers. 27/27.
- **Packaging** (no new features, same shape as 1.0.0's own final tier): `release/` refreshed from a stale 1.0.0 copy to the current 2.4.0 working copy, the in-app About panel and a new `RELEASE_NOTES.md` updated to describe the real v2 feature set, and cross-engine verification repeated on both the working copy and the packaged release copy. Chromium and Firefox both came back 26/26 on both copies. WebKit had to be skipped outright this time — its Playwright browser can't launch on this particular WSL host at all (missing system libraries, no elevated access available to install them). That's an environment limitation specific to this machine, not a finding about the product, and it's the same category of gap 1.0.0's own summary already flagged: no independent way to hand-verify that engine here either.

**On the two real bugs, the thing most worth sitting with.** Both v2 Tier 3 and v2 Tier 4 shipped a clean, fully-passing automated verification run that turned out to be hiding a genuinely broken user-facing interaction. Neither was a logic bug in the ordinary sense, the paste thresholds were correct as written, and the drag-and-drop code existed and ran. Both were gaps in what the *test* actually exercised: a UI state (a lingering toast, a focus context) the test never put the app into, and a DOM interaction (an actual pointer drag) the test bypassed by calling the underlying function directly instead of simulating it. Independent, in-browser manual testing caught both; re-running the same automated suite harder would have found neither. 1.0.0's own retrospective already noted that multi-engine testing is "for lifecycle races, not the screenshot gallery" — v2's finding is that lesson's sibling: a green suite only tells you the paths it actually walked were fine, not that the paths a real hand and mouse take are the same ones.

**A smaller, stranger finding, worth keeping on its own merits.** Partway through v2 Tier 3's re-verification, a manual test that looked like it was still failing turned out to be a units mismatch, not a bug. A person estimating whether pasted text will clear a character-count threshold tends to eyeball it by word count, or by how long a paragraph "feels," not literal character count, and the gap between those two is larger than it feels: an ordinary 149-word paragraph is routinely 900–1,000+ characters, comfortably clearing a 900-character cutoff despite reading as short. Worth remembering for any future feature gated on a character count: don't estimate test input by eye, get an exact count first, or use a purpose-built string of a known exact length.

## What v2 adds to "if a sibling app got built tomorrow"

1.0.0's original five-point list (further up this document) still holds without correction. Two more, specific to what v2 actually hit:

6. A verify script that moves state by calling an internal function directly is testing the data model, not the product. If a feature lives behind a real user gesture, drag, a multi-step toast, focus-dependent behavior, the test has to go through the DOM the same way a hand would, or a completely broken interaction can still report clean.
7. When a threshold is measured in characters, whoever manually tests it needs an exact count, not an eyeballed guess. The gap between "how long text feels" and its literal character count is large enough on its own to manufacture a false bug report.

## A process note the outside account adds, v2 edition

Every one of v2's five tiers, plus the packaging pass, went through the same two-step check before being called done: the tool's own automated pass, then a separate independent manual pass in a real browser, deliberately kept apart from trusting the first one. That second step is what actually caught both real bugs above; a clean automated report alone would have shipped both of them. Worth naming plainly rather than treating as incidental: for this project, independent verification wasn't a formality layered on top of the real work, it *was* where two of the six shipped tiers' real defects were actually found.

---

## Bottom line, in both voices

**The agent's:** Whiteboard 1.0.0 was less interesting as "four templates" than as a small proof that architecture discipline, honest `file://` constraints, and real-browser verification can produce something you can put on a USB stick. Five more tiers later, the same discipline is what let v2 grow to five templates, cross-board search, a command palette, content-aware paste routing, and full undo-grade history without a rewrite: extend the shell's contract, never fork it, and treat a clean automated pass as a claim to be checked, not a verdict to trust.

**The outside view:** the value of a detailed, honest write-up doesn't come from the parts that went smoothly, it comes from naming the assumptions that turned out wrong, the tests that needed a second look, and the gaps nobody could fully close. 1.0.0 had one of those (the WebKit lifecycle race, caught only by insisting on a second engine). v2 had two, caught only by insisting on a second pass of human hands after the automation said done. That's not a coincidence, it's the same discipline applied twice, and it's exactly what makes it possible for someone else to move faster on their own similar build. This document, in both its voices, across both versions, is that, paid forward.

## A cold read from the agent that built this (2026-07-13)

Everything above was written close to when the work happened. This section is different on purpose: it's Grok 4.5 looking at the finished `apps/whiteboard/` tree with no memory of building any of it, days later, asked to react honestly to its own past work as a stranger would, and to keep verified claims clearly separate from inference. Not a recollection, a fresh read.

**First attempt, and why it doesn't appear here.** The first run pointed the same prompt at this repo, plus a GitHub URL, and got back several confident, specific-sounding "gaps": `DEV_GUIDE.md` supposedly still titled 1.0.0 with no Kanban documented, a missing `STATUS-v2-packaging-final.md`, `PROJECT_SUMMARY.md` supposedly a 1.0-only retrospective. All three were checked directly against the live repo and were wrong. What actually happened: the agent found a folder on local disk from an earlier working session, matching filenames, genuinely stale relative to this repo, and read that instead of the repo it was told to use, despite a URL to the real thing sitting right in the prompt. A local folder that looks like the thing it's being asked about outranked a URL, silently.

The fix wasn't more prompt persuasion, it was removing the local folder's ability to compete at all: clone the actual published repo fresh into an isolated scratch folder immediately before reading, tell the agent explicitly to ignore any other local copy anywhere on disk even if filenames match, and add an explicit boundary that this repo is a monorepo and only `apps/whiteboard/` is in scope. That second run is what follows below. Worth keeping on record: an evidence-based reflection is only as good as the evidence it's actually reading, and a coding agent with local filesystem access will trust disk over a URL by default.

**Cold read (2026-07-13, second attempt, verified against the live repo).**

I opened `apps/whiteboard/` cold, only that tree, and read `whiteboard.html`, the two guides, `PROJECT_SUMMARY.md`, and the `build-process/` archive (scripts, JSON reports, packaging status). I am not treating any other path as source of truth, and I do not claim to remember building this.

What stands out first is how much product is still one file. The root `whiteboard.html` is 7,357 lines, declares `APP_VERSION = '2.4.0'`, and registers five templates at the bottom: Paint, Notepad, Snip (UI name Annotate), Wordpad, Kanban. That matches `VERSION`, the user guide, and the packaging report's template list. I also checked hashes: root `whiteboard.html` and `releases/v2.4.0/whiteboard.html` are byte-identical (`dfa58231...`). The packaging JSON records the same SHA for working and release copies and claims 26/26 on Chromium and Firefox for both targets (104 passed, 0 failed, 2 WebKit skips). Those are facts on disk, not impressions.

Architecturally, the shell/template split looks deliberate and still load-bearing. The dev guide's contract (`register → create(ctx) → setState / mount / renderTools / getState / unmount`) matches what I can see in source for Kanban at ~6693 and the five `TemplateRegistry.register(...)` calls around 7301-7305. Storage is still keyed `whiteboard-app-v2` (with a legacy v1 key and a quarantine key). History and trash live as additive fields on the same store rather than a v3 migration; the constants the guide names (`HISTORY_MAX_SNAPSHOTS = 10`, 2-minute min interval, 7-day trash, soft ~2.5M chars, 5MB quota estimate) are present in the file at the cited lines. That is a clever, conservative choice if the goal was "ship more without stranding saves," and it is verifiable in code, not only in prose.

What feels unusual, in a good way, is the amount of honesty around verification failure modes. The changelog and project summary both describe green automated passes that still hid real UX bugs (stale paste toasts; Kanban drag tested via internal state instead of a real pointer gesture). In the source I can see supporting residue: `pasteRouteCharCount` with explicit 280/900 character routing, and a comment on Kanban that drag is pointer-based because Firefox will not start HTML5 drag from cards that are mostly textareas. The packaging suite's result names include "Kanban pointer drag moved card to In Progress," which at least shows the later tests claim to exercise gesture, not only data model. I cannot re-run those browsers here from this note alone, so the *reports* say pass; I am not independently re-proving the interactions.

What I would flag or question:

**Stale docs layered on a live file.** The first half of `DEV_GUIDE.md` still talks about a File menu, four boot registrations, and v1-era line numbers. The guide itself warns about drift and then has a verified v2 appendix. User-facing docs correctly say there is no File menu and that export/trash live in the command palette. The split is intentional, but if I were onboarding, I would treat anything above the "v2 additions" section as historical architecture, not a current map.

**Packaging path drift.** `STATUS-v2-packaging-final.md` still talked about `release/` and `verify-v2-package.mjs` against that layout. In *this* tree there was no `release/` directory; shipped snapshots live under `releases/v1.0.0/` and `releases/v2.4.0/`. That looked like a rename after the status note was written, or status that wasn't fully retargeted. Verified: the SHA identity story still holds for `releases/v2.4.0/`.

**Report provenance vs. this clone.** Every v2 tier JSON I opened still points `file` at `/home/jp/grok-build-projects/whiteboard-app/whiteboard.html` and versions step `2.0.0 → 2.1.0 → 2.2.0 → 2.3.0 → 2.4.0`. So these reports are artifacts from another working path, archived here after the fact. That does not make the pass counts false; it means the evidence is "we ran this elsewhere and copied the results in," which the build-process README essentially admits.

**Overwrite hygiene they already know is broken.** The archive itself says pre-fix tier-3 and tier-4 reports were overwritten by fix passes, and that only the final packaging STATUS survives under a reused name. `PROJECT_SUMMARY` recommends versioned report filenames and then documents doing the same overwrite again. **WebKit is still a gap.** Packaging status and the JSON agree WebKit was skipped for missing shared libraries on the WSL host. v1's narrative claimed a real WebKit lifecycle bug that was fixed; v2 packaging never re-proved WebKit on this machine. That is an environment limitation as documented, and also a remaining hole if Safari/`file://` matters.

**`package.json` was thinner than the README/status implied.** It only defined `npm run verify` → `verify.mjs`. The v2 scripts existed as files, but the npm script table wasn't updated to match the `npm run verify:v2-tier5` style commands mentioned in STATUS. Minor, but it was a small crack between "documented verify entrypoints" and what `package.json` actually wired up.

**Product tradeoffs I can see, and ones I only hear about.** In-file choices that look deliberate: offline-first (no network in the packaging checks), paste/drop as the real Annotate path with capture treated as bonus, import appends, fixed template type per board, PNG-in-localStorage with quota meter and snapshot trim. Things I am *not* treating as proven from this pass alone: that multi-tab safety or corrupt-state quarantine feel right under real abuse, that the Mullvad canvas stripe "known issue" still reproduces, or how painful 7k lines of inlined CSS/JS is to edit day to day; the project summary frets about single-file scale, and looking at the file I share the worry, but that is judgment, not a measured claim.

One more inference, clearly marked as such: the project summary's two-voice structure (agent vs outside verifier) and the habit of writing failure stories into `CHANGELOG`/`PROJECT_SUMMARY` feel like process designed for a future cold reader, which is slightly meta, given that is what I am doing. Whether that process was overbuilt for a clipboard app or exactly what kept a five-tier v2 from forking the shell, I can only guess. What I *can* verify is that the shell contract appears intact after that growth, and that the test archive is unusually candid about when green meant "we tested the wrong thing."

I would not rewrite the product from this reading. If I touched anything next, it would be documentation/layout consistency (`release/` vs `releases/`, File-menu leftovers in the old half of the dev guide), report naming so the interesting failure evidence survives, and a real WebKit/Safari pass somewhere that is not this WSL host, none of which I did; this is observation only.

**What came of it.** Two of the flagged issues were checked directly against the live repo and were real: `build-process/STATUS-v2-packaging-final.md` described a `release/` folder that no longer exists (this repo now ships `releases/v1.0.0/` and `releases/v2.4.0/`, from this same session's earlier restructure), and `build-process/package.json` only wired up the original `verify` script, not any of the five `verify:v2-tier*` or `verify:v2-package` commands the status doc tells people to run.
